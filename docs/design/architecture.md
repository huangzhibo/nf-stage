# 百趣生信流程阶段归档与复用 — 目标架构

## 1. 文档定位

本文档定义百趣生信 Nextflow 流程"阶段归档与复用"能力的目标架构，作为内部架构决策记录。

相关文档：

- [流程开发规范](../pipeline-guide.md)
- [平台接入规范](../platform-guide.md)

## 2. 目标

本方案要解决的不是原生 `-resume`，也不是恢复旧 `work/`，而是：

- 删除历史 `work/` 后，仍可复用少量已保存阶段继续分析
- 只长期保留真正需要跨阶段复用或最终交付的文件，尽量节省存储
- 用户修改参数后，系统能自动判断哪些阶段可以跳过、哪些阶段必须重跑
- 支持手动指定某个已保存阶段，从其下游继续分析

## 3. 核心结论

### 3.1 阶段边界

- `named workflow` 是唯一阶段边界
- 阶段之间只通过 `take:` / `emit:` 连接
- MVP 中，stage 不允许直接依赖阶段外 `process` 输出

### 3.2 长期保留内容

- 阶段长期保留内容只包括：`stage.json + 阶段 emit`
- 阶段内部未进入 `emit:` 的文件默认不进入长期归档，可随 `work` 删除
- `workflow outputs` 和 `fan-in` 只能依赖阶段 `emit:` 或其下游显式输出

### 3.3 恢复逻辑

- 运行时在阶段边界即时判断是否复用
- 若某阶段历史归档兼容且完整，则直接复用该阶段，并停止继续追溯该分支上游
- 若某阶段不可复用，则该阶段及依赖其输出的下游阶段需要执行
- 只有真正保留阶段资产的阶段参与恢复判断；非归档语义阶段不参与阶段恢复逻辑
- 不需要独立 `plan` 阶段，也不要求两次运行

### 3.4 资产模型

- 命中相同 `compatibility_digest` 时，直接复用已有阶段资产
- 只有未命中时，才创建新的阶段资产目录
- `workflow_id` 不是路径主键，只用于审计和引用关系

## 4. 硬约束

以下约束应作为评审和验收基线：

1. 阶段内部未进入 `emit:` 的大文件可随 `work` 删除，且不影响从已保存阶段继续运行
2. 历史归档中若不存在与当前阶段兼容性摘要匹配的归档，则必须重跑该阶段以及依赖其输出的下游阶段
3. 流程可以只定义少量阶段；每个阶段内部允许包含很多普通 `process`
4. MVP 中，stage 的必要输入只允许来自上游 stage `emit:` 或稳定外部输入
5. 命中相同 `compatibility_digest` 时，应直接复用已有阶段资产，不再写新的资产目录

## 5. 核心术语

### 5.1 阶段

阶段是一个 `named workflow`，表示一组有明确输入、稳定输出、值得保存的流程逻辑。

### 5.2 阶段归档

阶段归档是某次阶段执行成功后保存的不可变阶段资产，包含：

- `stage.json`
- 阶段 `emit:` 对应的复用产物
- 兼容性判断所需 metadata
- 审计信息

### 5.3 阶段复用产物

阶段复用产物是后续续跑、`workflow outputs` 或 `fan-in` 所依赖的稳定文件或稳定引用集合，默认等同于阶段 `emit:` 的命名输出。

### 5.4 兼容性摘要

当前主线只保留两个机器摘要：

- `input_digest`：描述阶段实际消费的关键输入
- `compatibility_digest`：用于判断历史资产能否被当前运行直接复用

二者均由运行时自动生成。

## 6. 阶段结构要求

### 6.1 唯一阶段边界

唯一阶段边界是 `named workflow`。

### 6.2 take / emit 规则

- `take:` 表示该阶段全部输入
- `emit:` 表示该阶段全部命名输出
- `emit:` 的全部命名输出，视为该阶段长期保留的阶段复用产物
- 阶段内部 `process` 产生但未进入 `emit:` 的文件，不属于阶段归档
- `emit:` 允许是文件或目录
- 目录型 `emit` 若包含超过 `1000` 个文件，MVP 中运行时应直接报错
- 若目录型输出预计包含大量小文件，流程作者应在阶段内部先收敛

### 6.3 普通 process 规则

MVP 中：

- 普通 `process` 可以大量存在，但应位于阶段内部
- stage 不允许直接依赖阶段外普通 `process` 的输出
- 阶段输入只来自上游阶段 `emit:` 或稳定外部输入
- 非归档语义步骤只作为代码组织或进度展示语义，不参与恢复判断

## 7. 职责边界

### 7.1 平台负责

- 阶段归档定位
- 执行入口管理
- 恢复审计记录

### 7.2 运行时负责

- 阶段兼容性判断
- 阶段跳过与继续执行
- 读取和写入阶段归档

### 7.3 流程作者负责

- 定义少量高价值阶段
- 保证阶段 `emit:` 是长期保留、可复用、可交付的稳定输出
- 不应把仅用于前端展示的语义步骤混入阶段恢复判断

## 8. 非目标

- 不恢复旧 `work/`
- 不做 task 级恢复
- 不引入新的阶段 DSL
- 不要求流程作者显式维护复杂版本号体系
- 不要求平台理解"哪个参数影响哪个阶段"
- 不让非归档语义阶段参与恢复判断

## 9. 多集群策略

当前主推荐策略不是分阶段跨集群编排，而是：

- 重跑优先回原集群
- 新项目再按负载调度
- 只有在原集群明显不合适时，才考虑后续阶段边界迁移

阶段边界迁移保留为后续增强能力，不纳入当前主线。nf-stage 插件只负责阶段资产产出和恢复判断，不负责集群选择。
